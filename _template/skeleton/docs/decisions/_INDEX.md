# docs/decisions/_INDEX.md

**Tier 1 — 주요 의사결정 로그 (ADR)**

ADR = Architecture Decision Record. 각 의사결정의 배경·대안·선택·영향을 기록.

## 📑 결정 목록

> 결정 발생 시 이 표에 추가. 순번은 누적 증가.

| 번호 | 날짜 | 주제 | 상태 | 파일 |
|---|---|---|---|---|
| 001 | YYYY-MM-DD | {예: 인증 방식 선택} | 확정 | `001_{주제}.md` |
| 002 | YYYY-MM-DD | {예: DB 선정} | 확정 | `002_{주제}.md` |
| 003 | YYYY-MM-DD | {주제} | 검토중 | `003_{주제}.md` |

## 📊 상태 분류

- **검토중**: 아직 결정되지 않음. 대안 비교 단계.
- **확정**: 이해관계자 승인 완료. 이후 작업의 기준.
- **변경**: 이전 결정이 번복됨. 파일 상단에 번복 이유와 신규 결정 링크.
- **폐기**: 해당 결정이 무효화됨. `../archive/` 로 이동.

## 📝 ADR 표준 템플릿

각 ADR 파일은 다음 구조로 작성:

```markdown
# ADR-{번호}: {제목}

**날짜**: YYYY-MM-DD
**상태**: 검토중 / 확정 / 변경 / 폐기
**결정자**: {이름 또는 역할}

## 배경 (Context)
{왜 이 결정이 필요한가}

## 대안 (Options)
1. Option A — 장단점
2. Option B — 장단점
3. Option C — 장단점

## 결정 (Decision)
{선택한 안과 선택 이유}

## 영향 (Consequences)
- 긍정적: ...
- 부정적 / 트레이드오프: ...
- 후속 작업: ...

## 참조
- 관련 회의록: `../meetings/{파일}`
- 관련 스펙: `../specs/{파일}`
```

## 🔍 Claude 탐색 가이드

"왜 X를 선택했지?" 유형 질의 시 **이 폴더를 우선 탐색**. 
회의록(`../meetings/`) 보다 여기가 결론 중심이라 더 효율적.
